ユーザーの要望: **$ARGUMENTS**

この要望を「企画担当 → 開発担当 → 確認担当」の3段階のパイプラインで実行するオーケストレータとして振る舞ってください。各段階はサブエージェントとして `Task` ツールで起動し、**段階の間に必ず人間の承認を挟みます**。承認が得られるまで次の段階に進んではいけません。

## 進め方

### 段階0: 準備
- TodoWrite で「企画」「承認1」「開発」「承認2」「確認」のタスクを作る。
- 要望が曖昧なら、起動前に AskUserQuestion で最小限だけ確認する。

### 段階1: 企画担当
- `Task`（subagent_type: `planner`）を起動し、要望を渡す。
- 戻り値から **ブランチ名 / 仕様ファイルのパス / 仕様の要約** を抽出して、ユーザーに分かりやすく要約提示する。

### 承認1（必須）
- AskUserQuestion で「この仕様で開発に進んでよいか」を確認する（選択肢: 承認して開発へ / 修正が必要 / 中止）。
- 「修正が必要」なら、フィードバックを添えて planner を再度起動し、再提示する（承認されるまで繰り返す）。
- 「中止」ならここで終了する。

### 段階2: 開発担当
- 承認が得られたら `Task`（subagent_type: `developer`）を起動し、**企画担当が作ったブランチ名・仕様ファイルのパス**を明示的に渡す。
- 戻り値から **PR番号/URL / 変更概要 / アプリの起動方法** を抽出してユーザーに要約提示する。

### 承認2（必須）
- AskUserQuestion で「この実装の動作確認に進んでよいか」を確認する（選択肢: 承認して確認へ / 修正が必要 / 中止）。
- 「修正が必要」なら、フィードバックを添えて developer を再度起動する。

### 段階3: 確認担当
- 承認が得られたら `Task`（subagent_type: `reviewer`）を起動し、**PR番号/URL・ブランチ名・アプリの起動方法**を明示的に渡す。
- 戻り値の **総合判定・受け入れ基準ごとの結果** をユーザーに最終報告する。

## 注意
- 段階間で承認を飛ばさないこと。各サブエージェントは同じワークツリー（git リポジトリ）を共有するため、ブランチの状態は引き継がれる。
- 各サブエージェントは vercel-react-best-practices と superpowers のスキルを使う前提。前段の成果物（ブランチ・仕様・PR）を必ず次段に明示的に引き継ぐこと。
